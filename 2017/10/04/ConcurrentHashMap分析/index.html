concurrentHashMap初始化主要是初始化sizeCtl字段为2的幂，调用的方法为tableSizeFor，简单说一下这个方法的实现，先放出方法的源代码

说明如下：

在二进制中，一个数从左向右最高有效位一定是1，看下下列的二进制数体会一下就自然而然明白了

十进制表示 二进制表示
1 1
2 10
3 11
4 100

7 111

那这段代码n |= n >>> 1;翻译一下其意思就是将最高为的 1右移一位后形成的新数与原数做或运算，那会是什么样的数呢，不管其数值是多少，可以肯定的是最高 2 位必定是 11, 所以接下来的代码n |= n >>> 2;就好理解了，就是形成的新数n的最高4位一定是1111。依次类推后面的代码，那么为什么到16就结束了，因为右移16位后进行异或时，最大可以让其右32位的1, 而 JAVA 中的 int 类型长度位 32 位，所以 16 就可以了，所以通过上述移动以后一个数如果是正数且不大于Integer.MAX_VALUE, 当进行上述位移后，数值n的二进制位上每个数值都变成了 1 ，然后进行 + 1 操作就找打了比他大且离他最近的2的幂的数 。有木有很精髓呢？

put

put 函数最终调用了 putVal(key,value) 函数，这个函数是通过一个无限循环函数加中断函数的形式完成结点的插入操作的，这里主要运用到了cas函数进行操作，然后对 Node[] table 中的某个 Node 结点加同步锁 synchronized实现并发插入的。这个函数在插入时主要考虑了如下几个问题

  1. 当前Map是否完成 Node[] table 的初始化？
  2. 当前Map是否正在进行扩容？
  3. 当前Map是否需要进行数据存储格式转化(链表->树)？
  4. 插入完成后是否需要扩容？

扩容

concurrentHashMap 中的扩容实现是非常精髓的，扩容的核心逻辑如下

是通过开辟一个新的数组空间, 其为原来的2倍，将老的数组空间上的数据结点拷贝到新的数组空间上

单线程实现上面的逻辑是非常简单的，但是多线程就比较麻烦了，因为要考虑有可能多线程之间并发处理同一个数组空间结点，concurrentHashMap是如何解决这个问题的呢？ 其实核心思想还是比较简单的，就是不让多线程同时处理数组中的同一个结点，而是给每个参与扩容的线程固定的结点数，那么如何做的呢，简单概括如下

从数据的最大结点开始给每个参与扩容的线程数固定的数组空间，每个线程处理的数据空间大小根据CPU核数决定，如果CPU为单核，则每个线程处理的数组空间大小为max(16, n), n代表数组大小，如果为多核，则每个线程处理的数组空间大小为max(16, n >>> 3 /NCPU), n代表数组大小，NCPU代表CPU核数，同时说明单可用CPU内核数为1时，只有单线程参与扩容
